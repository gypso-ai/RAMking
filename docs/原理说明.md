## 1. 项目定位

RAMking 是一个**轻量级的 C 内存安全包装层**。它不替换系统分配器，而是在调用标准 `malloc` / `calloc` / `realloc` / `free` 的同时，将每一次分配的元信息记录在一张**静态表**中，从而实现：

- **重复释放（double-free）检测**
- **释放未登记指针（unregistered free）检测**
- **当前存活分配数量统计**
- **历史峰值统计**
- **全量分配历史转储**

整个模块 **不为自身的记账结构调用 `malloc`**，因此不存在递归分配问题，也不会因堆耗尽而使记账失效。

## 3. 记录表（Record Table）

### 数据结构

```c
typedef struct {
    void        *ptr;    /* 已分配的指针                  */
    size_t       size;   /* 请求分配的字节数              */
    int          freed;  /* 0 = 存活, 1 = 已释放          */
    unsigned int seq;    /* 单调递增的分配序号            */
} SafeAllocRecord;

static SafeAllocRecord g_records[SAFE_ALLOC_MAX_RECORDS];
```

记录表是一个**编译期固定大小的静态数组**（默认 1024 槽位），不依赖堆内存。

### 槽位生命周期

```
空槽（freed=1 或从未写入）
        │
        │  register_alloc()
        ▼
    已分配（freed=0）
        │
        │  safe_free() 或 safe_realloc(ptr, new_ptr≠ptr)
        ▼
    已释放（freed=1）—— 槽位可被下一次分配复用
```

### 槽位复用策略

`find_free_slot()` 优先扫描 `freed == 1` 的槽位。若无可用的已释放槽，则尝试将高水位线（`g_used`）向后扩展一位。若数组已满，则返回 `-1`，并通过日志处理器发出警告；此时分配仍然成功，但该指针**不被跟踪**。

### 查找算法

| 函数 | 行为 |
|---|---|
| `find_alive(ptr)` | 线性扫描，返回第一个 `ptr` 匹配且 `freed==0` 的索引 |
| `find_any(ptr)` | 线性扫描，返回任意（包括已释放）匹配记录的索引 |
| `find_free_slot()` | 线性扫描，返回可复用的空槽索引 |

所有查找均为 **O(n)**，n ≤ `SAFE_ALLOC_MAX_RECORDS`。这对于嵌入式或调试场景已经足够；若需更高性能，可将记录表改为哈希表。

## 4. 核心分配接口原理

### 4.1 `safe_malloc`

```
safe_malloc(size)
    │
    ├─ 调用 malloc(size)
    │      失败 → 通过日志处理器输出 ERROR，返回 NULL
    │
    └─ register_alloc(ptr, size)
           ├─ find_free_slot() 获取槽位
           ├─ 写入 ptr / size / freed=0 / seq++
           ├─ g_alive++, g_total_allocs++
           └─ 更新 g_peak（若 g_alive > g_peak）
```

### 4.2 `safe_calloc`

与 `safe_malloc` 相同，区别在于底层调用 `calloc(nmemb, size)`（保证零初始化），并以 `nmemb * size` 作为记录的 `size` 字段。

### 4.3 `safe_realloc`

`realloc` 的语义较复杂，共有四种路径：

```
safe_realloc(ptr, new_size)
    │
    ├─ ptr == NULL  → 转发给 safe_malloc(new_size)
    │
    ├─ new_size == 0 → 转发给 safe_free(ptr)，返回 NULL
    │
    ├─ 调用 realloc(ptr, new_size)
    │      失败 → 输出 ERROR，原有记录保持不变，返回 NULL
    │
    └─ 成功，分两种情况更新记录表：
           ├─ new_ptr == ptr（指针未移动）→ 仅更新 size 字段
           └─ new_ptr != ptr（指针已移动）
                  ├─ 旧记录：freed=1, g_alive--, g_total_frees++
                  └─ 注册新指针：register_alloc(new_ptr, new_size)
```

> **"指针移动"原子性**：旧记录的标记与新记录的写入在同一函数调用内完成，不存在中间态泄漏。

### 4.4 `safe_free`

```
safe_free(ptr)
    │
    ├─ ptr == NULL → 直接返回（符合 C 标准）
    │
    ├─ find_alive(ptr)
    │      找不到 → find_any(ptr)
    │                ├─ 找到且 freed==1 → 输出"double-free"警告，不调用 free()
    │                └─ 找不到          → 输出"unregistered ptr"警告，不调用 free()
    │
    └─ 找到存活记录
           ├─ g_records[idx].freed = 1
           ├─ g_alive--, g_total_frees++
           └─ 调用 free(ptr)
```

> **安全语义**：对已释放或未登记的指针，`safe_free` **拒绝调用 `free()`**，从根本上杜绝了重复释放导致的未定义行为。

## 5. 诊断与统计接口原理

### 统计计数器

所有统计字段均为模块内部的静态 `unsigned int` 变量，由 `safe_alloc_reset()` 清零。

| 接口 | 返回值 |
|---|---|
| `safe_alloc_alive_count()` | `g_alive`：当前存活（未释放）的分配数 |
| `safe_alloc_peak_count()` | `g_peak`：历史最高同时存活数 |
| `safe_alloc_total_allocs()` | `g_total_allocs`：累计成功分配次数 |
| `safe_alloc_total_frees()` | `g_total_frees`：累计成功释放次数 |

### 转储接口

- `safe_alloc_dump_alive()`：遍历 `g_records[0..g_used-1]`，仅打印 `freed == 0` 的记录。
- `safe_alloc_dump_all()`：打印所有记录，包括已释放的，并附带统计摘要。

两者均通过**可替换日志处理器**输出，不再直接绑定 `stderr`。

### `safe_alloc_get_records`

将 `g_records[0..g_used-1]` 按值复制到调用方提供的数组中。调用方可自行过滤和分析，无需依赖文本输出。

### `safe_alloc_reset`

```c
memset(g_records, 0, sizeof(g_records));
g_used = g_seq = g_alive = g_peak = g_total_allocs = g_total_frees = 0;
```

将整张表清零并重置所有计数器。**不释放任何堆内存**，仅用于单元测试的隔离（每个测试用例独立从干净状态开始）。

## 6. 可替换日志处理器

### 设计动机

原始实现将所有警告和错误直接写入 `stderr`。在以下场景中，这种硬编码方式不满足需求：

- **单元测试**：需要断言是否产生了特定警告，而不是让警告淹没测试输出。
- **GUI / 嵌入式**：没有 `stderr`，或需要将日志路由到自定义渠道。
- **日志框架集成**：希望统一使用项目的日志库（如 `spdlog`、`zlog` 等）。

### 接口设计

```c
/* 严重级别 */
typedef enum {
    SAFE_ALLOC_LOG_WARNING = 0,  /* 非致命：双重释放、未登记释放等 */
    SAFE_ALLOC_LOG_ERROR   = 1   /* 严重：分配失败                 */
} SafeAllocLogLevel;

/* 回调类型 */
typedef void (*SafeAllocLogFn)(SafeAllocLogLevel level, const char *msg);

/* 安装自定义处理器；传入 NULL 恢复默认行为 */
void safe_alloc_set_log_handler(SafeAllocLogFn fn);
```

### 内部实现原理

1. 模块内部维护一个静态函数指针 `g_log_fn`，初始值指向内置的 `default_log_handler`（调用 `fprintf(stderr, ...)`）。
2. 所有原本直接调用 `fprintf(stderr, ...)` 的位置，改为调用内部辅助函数 `safe_log(level, fmt, ...)`。
3. `safe_log` 使用 `vsnprintf` 将格式化消息写入一个固定大小的栈缓冲区（512 字节），然后调用 `g_log_fn`。
4. `safe_alloc_set_log_handler(fn)` 仅修改 `g_log_fn` 指针；传入 `NULL` 时自动还原为 `default_log_handler`。

### 典型用法

```c
/* 测试中捕获警告，不让其污染测试输出 */
static int g_warn_count = 0;

static void my_log_handler(SafeAllocLogLevel level, const char *msg)
{
    (void)msg;
    if (level == SAFE_ALLOC_LOG_WARNING) ++g_warn_count;
}

void test_double_free_warning(void)
{
    safe_alloc_set_log_handler(my_log_handler);
    void *p = safe_malloc(8);
    safe_free(p);
    safe_free(p);  /* 触发警告 */
    assert(g_warn_count == 1);
    safe_alloc_set_log_handler(NULL);  /* 恢复默认 */
}
```

## 7. 设计权衡与限制

| 方面 | 当前设计 | 说明 |
|---|---|---|
| 记录表大小 | 编译期固定 | 避免动态分配，适合嵌入式；最大并发分配数受限 |
| 查找复杂度 | O(n) 线性扫描 | 简单可靠；n ≤ 1024 时性能充足 |
| 线程安全 | 无锁 | 单线程或外部加锁场景使用；多线程需自行加锁 |
| 内存泄漏检测 | 通过 `dump_alive` 或 `get_records` 手动触发 | 不自动在程序退出时报告 |
| 溢出处理 | 记录表满时仍分配，但不跟踪 | 保证分配不失败；发出 WARNING |

## 8. 配置与扩展

### 修改最大记录数

在编译时通过 `-D` 覆盖默认值（1024）：

```sh
make CFLAGS="-DSAFE_ALLOC_MAX_RECORDS=4096 -Wall -Wextra -std=c99 -g"
```

或在包含头文件之前定义：

```c
#define SAFE_ALLOC_MAX_RECORDS 4096
#include "safe_alloc.h"
```

### 集成到现有项目

只需将 `include/safe_alloc.h` 和 `src/safe_alloc.c` 复制到目标项目，无外部依赖（仅使用 C99 标准库）。
